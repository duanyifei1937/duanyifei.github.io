发现一个很奇怪的现象：

大家都在开发一个又一个系统：aa平台、bb平台

大多数都在为了开发一个系统而开发一个系统；

其实见过几个就会知道，都大同小异而已：前端展示、mysql、一堆增删改查的功能；

是在不敢恭维做的好，定制化程度高；

甚至都比不上一些开箱即用的开源工具；

一些基本功能都没有，还有必要做这一个有一个系统吗？

口说不凭，现实举例；并且抱怨别人做的垃圾也不好，说说自己的想法、并付出实践：

  • nginx如何上线：

    • 同志们都想的很复杂，自己要做版本管理工具、涉及一些复杂的部分怎么去调用、各种拆分location、各种引用location. 等等。

    • 却忽略了最简单也是最原始的需求：发布(一键发布)、回滚(记录发布历史方便回滚)；

    • Jenkins 就完全满足了这些基本需求，所以之前一直用的挺好；

    • z

    • 现在如何上线：

      • git commit;

      • git tag;

      • jumpserver准备命令：ansible-playbook nginx-conf-gitsync.yml -i inventory -e '{"branch":"dynamic-v20200827.2158", "actions":"update", "hosts": "nginx_dynamic"}'

        ansible-playbook nginx-conf-gitsync.yml -i inventory -e '{"branch":"dynamic-v20200827.1637", "actions":"rollback", "hosts": "nginx_dynamic"}'

        • 为了发一下，我需要

          • 登录jumpserver;
          • 贴new tag;
          • 登录机器查看old tag;
          • –list-hosts 确认发布机器；
          • 发布；

          这样传参太多: branch_tag + action + hosts;

    • 之前如何上线：

      • Git commit；
      • get commit id –> jenkins(不知道commit id 能不能跨branch, 验证下)；
      • host 写死；
      • 回滚直接找commit id;

    • 基于现在做如下修改：

      • 不打tag， 直接git check $commit_id;
      • 多个nginx 部署引用playbook;

  • 之后内容：

    • ansible 调用一个dynamic invertory; 暴露接口引用；
    • 如何记录历史方便一点？

简化nginx部署流程

之前部署方式

Branch tag

  • git master commit;

  • git create new tag;

  • jumpserver准备命令：

2

    ansible-playbook nginx-conf-gitsync.yml -i inventory -e '{"branch":"dynamic-v20200827.2158", "actions":"update", "hosts": "nginx_dynamic"}'
ansible-playbook nginx-conf-gitsync.yml -i inventory -e '{"branch":"dynamic-v20200827.1637", "actions":"rollback", "hosts": "nginx_dynamic"}'

* 需要传参： $branch_tag + $action + hosts;

简化后部署方式：

login jumpserver && cd /etc/ansible

ansible-playbook nginx-deploy.yml -i ./nginx-hosts -e hosts=nginx_testenv -e commitid="ff188281ccc6b2dec949d5971fe15afa984b1c97"

统一使用commit id (不在区分branch); 减少传参数；找history回滚；
