记录一些m3db使用过程中的一些问题

m3db的优点

M3db 比起 thanos 其实比较重，相当于维护了一套一致性db, 和 elasticsearch很像；

但m3db也有优于thanos的点：

  • 自定义事件粒度压缩(thanos只能超过40h generation 1min; >10d generation 5min; 并且无法调整, 在数据量过大需要即时压缩时，无法满足需求；)

m3db缺点

文档不完善、很多问题没有解决；

m3coordinator datapoint too far

m3coordinator往m3db里写数据，偶发性报错：datapoint too far;

报错后数据无法写入，只能restart prometheus得以解决；

Namespaces 中存在相关配置：

1
2

"bufferFuture": "2m",
"bufferPast": "2m",

m3db实时写入数据必须符合时间窗口范围：now() - bufferFuture < $metrics_time < now() + bufferPast;

在实际使用过程中发现，写入越来越延迟，最终超过该范围；(类似于ES log 写入延时)

优化过程中，主要针对于m3db的写性能，其实偏差；

m3coordinator 发往m3db自身有个queue, 当数据量过大(我们当前1300+ target; 80k samples/sec)就会有性能问题；

通过拆解target得以解决；

m3coordinator 带宽使用问题

如图，m3coor 与 prometheus 部署在同一台节点，prometheus拉取metrics 15Mbps, m3coor上传 1.56Gbps.

由于现在使用混合云，之间专线打通，带宽有限，必须清楚为什么上传的数据块 扩大了百倍？

查了下thanos 从sidecar的上传、下载也存在这种情况：大概40倍差距；

给出的解释：

  1. replica配置；
  2. prometheus metrics incoming are bukdled, m3coordinator 格式化拆分；
  3. compression问题(prometheus –> m3coordinator 高压缩编码， m3coordinator –> m3db 未使用数据压缩)；
  4. 额外增加的tag;

使用LZ4 未完成 https://github.com/m3db/m3/pull/2079

自定义retention time

thanos 可以实现unagg数据保留7d, 1min agg 数据保留15d, 5min 数据保留一个月；(节约存储成本 + 提高查询性能)

m3db没有该功能;

由于只能有一个unagg namespaces, 只能定义一个保留时间，不同prometheus 就必须保留相同时间；

尝试：写入间隔15s的数据但定义为agg数据，但写入报错；(不能用该歪门邪道方式)

即使使用同一个unagg namespace, 有没办法只保留超出规定时间的压缩数据(降样本数据)，删除掉为压缩数据？
